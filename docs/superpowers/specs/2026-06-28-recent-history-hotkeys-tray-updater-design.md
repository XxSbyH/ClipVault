# 最近历史快捷键、托盘交互与自动更新设计

## 背景

ClipVault 当前已经支持全局快捷键、固定内容快捷键、快速复制上一项/下一项历史、系统托盘菜单和手动版本发布。xxsby 希望新增三类能力：

- 在设置的快捷键页中配置最近十条历史记录的位置快捷键。
- 调整托盘图标交互：左键直接打开控制面板，右键显示菜单。
- 后续支持客户端启动时自动检测新版本，并在用户同意后自动更新。

本设计分两阶段执行。第一阶段实现最近十条历史快捷键和托盘交互；第二阶段实现自动更新，因为它会影响发布签名、Release 产物和安装包发布流程。

## 范围

第一阶段包含：

- 快捷键页新增“最近十个历史记录”分组。
- 位置 1 到 10 可独立配置快捷键，位置 1 表示最新一条真实复制历史，位置 2 表示次新，以此类推。
- 触发位置快捷键时，直接粘贴对应历史项内容到当前前台应用。
- 托盘左键打开并聚焦主控制面板。
- 托盘右键显示菜单，菜单移除“打开 ClipVault”项。

第二阶段包含：

- 集成 Tauri updater。
- 启动客户端时检查更新。
- 有更新时弹窗说明版本、更新内容和大小；用户同意后下载、安装并重启应用。
- 发布流程生成并上传 updater 需要的签名产物和 `latest.json`。

不包含：

- 不做用户自定义数据存储路径。
- 不修改 Windows 系统 `Win+V` 行为。
- 不做静默强制更新。
- 不把最近十条快捷键做成 `Ctrl+A+1` 这种按键序列。第一阶段只支持当前全局热键体系稳定支持的“修饰键 + 一个主键”组合，例如 `Ctrl+Alt+1`。

## 用户行为

### 最近十个历史记录快捷键

假设历史复制顺序为 `a, b, c, d, e`，其中 `e` 是最后复制的内容：

- 位置 1 触发时粘贴 `e`。
- 位置 2 触发时粘贴 `d`。
- 位置 5 触发时粘贴 `a`。
- 位置 6 到 10 因为没有对应历史项，触发后不粘贴，并显示 HUD 提示“最近历史不足”。

“最近”按实际创建时间倒序计算，忽略置顶优先级。这样即使旧内容被置顶，位置 1 仍然是最新复制的内容。

默认不预设这十个快捷键，避免抢占系统或常用软件快捷键。用户可在快捷键设置页逐项录制。推荐组合显示在说明文案中：`Ctrl+Alt+1` 到 `Ctrl+Alt+0`。

### 托盘图标

- 左键单击托盘图标：打开并聚焦主窗口。
- 右键单击托盘图标：显示菜单。
- 右键菜单包含：暂停/恢复监听、清空历史、设置、打开官网、退出。
- 因左键已经提供打开能力，右键菜单不再显示“打开 ClipVault”。

### 自动更新

客户端启动后检查更新。检查结果分三类：

- 无更新：不打扰用户。
- 有更新：显示版本号、发布时间和更新说明，用户确认后下载并安装。
- 检查失败：不阻塞应用启动，只记录日志；关于页可显示“检查更新失败，请稍后重试”。

更新安装完成后提示重启，用户确认后调用重启能力。Tauri updater 要求更新包签名验证，私钥不能进入仓库。

## 架构设计

### 最近历史快捷键模型

新增独立模型，不把十个位置硬塞进已有 `HotkeySettings`：

- Rust：`RecentHistoryHotkey`
- TypeScript：`RecentHistoryHotkey`

字段：

- `slot`: 1 到 10。
- `hotkey`: 快捷键字符串，空字符串表示未绑定。
- `enabled`: 是否启用。

持久化使用 SQLite `settings` 表的独立键：

- `recentHistoryHotkey1`
- `recentHistoryHotkey2`
- ...
- `recentHistoryHotkey10`

每个值保存 JSON：`{"hotkey":"Ctrl+Alt+1","enabled":true}`。这样不需要新增表，也与现有设置和热键存储模式一致。

### 快捷键注册

后端启动和热键更新时统一注册：

- 常规快捷键。
- 固定内容快捷键。
- 已启用且有 hotkey 的最近历史位置快捷键。

新增动作：

- `HotkeyAction::RecentHistorySlot(u8)`

触发后：

1. 根据 slot 取 `offset = slot - 1`。
2. 使用新的 repository 方法按 `created_at DESC, id DESC` 读取对应历史项，忽略置顶排序。
3. 调用现有粘贴路径写入剪贴板并模拟 `Ctrl+V`。
4. 成功后更新 use stats、历史 revision，并显示 HUD。
5. 历史不足或粘贴失败时显示轻量 HUD/日志，不修改 use stats。

### 冲突检测

最近历史快捷键需要参与全部冲突检测：

- 不允许与常规快捷键重复。
- 不允许与固定内容快捷键重复。
- 不允许与其他最近历史位置重复。
- 系统占用检测仍沿用 `checkHotkeyAvailable`，占用时允许用户确认后保存，保持现有快捷键体验。

### 前端设置页

在快捷键页中新增一个分组，放在“历史快速复制”和“固定快捷内容”之间：

- 标题：`最近十个历史记录`
- 说明：`为最近 1-10 条历史绑定快捷键，触发后直接粘贴对应位置的内容。`
- 每一行显示：
  - `位置 1（最新）`
  - `位置 2`
  - ...
  - `位置 10`
  - 快捷键录制按钮
  - 启用开关

录制交互复用当前常规快捷键录制逻辑。保存时调用新增 API，而不是复用 `updateHotkeys`。

### 自动更新发布机制

第二阶段采用 Tauri 官方 updater 插件。Tauri 官方文档说明 updater 支持检查、下载、安装和重启，并要求签名验证；配置需要 `createUpdaterArtifacts`、公钥和更新端点。

推荐端点：

```text
https://github.com/XxSbyH/ClipVault/releases/latest/download/latest.json
```

发布时需要：

- 生成 updater 签名密钥。
- 私钥保存在发布环境或 xxsby 本机安全位置，不提交仓库。
- 公钥写入 `tauri.conf.json`。
- 打包时生成 updater artifacts。
- GitHub Release 上传安装包、签名和 `latest.json`。

### CI 自动签名发布方案

后续可以把自动更新发布迁移到 GitHub Actions 或其他 CI，避免每次在本机交互输入私钥密码。推荐做法：

- 将 updater 私钥内容保存为 CI secret：`TAURI_SIGNING_PRIVATE_KEY`。
- 将 updater 私钥密码保存为 CI secret：`TAURI_SIGNING_PRIVATE_KEY_PASSWORD`。
- Release workflow 只在受保护的 tag 或手动审批后触发，避免普通分支构建接触签名密钥。
- CI 构建时设置 Rust 环境、pnpm store 和 `CARGO_BUILD_JOBS=1`，再执行 `pnpm build`。
- CI 只上传 `ClipVault_*_windows_x64_setup.exe`、对应签名文件和 `latest.json` 到 GitHub Release，不上传私钥或包含私钥的日志。
- 私钥一旦丢失或密码遗忘，已安装客户端无法继续验证后续更新；私钥一旦泄露，需要停止使用旧 updater 通道并重新设计迁移方案。

## 测试策略

第一阶段自动化测试：

- Rust model 默认值和序列化测试。
- Repository 最近历史 offset 测试，验证忽略置顶排序。
- 最近历史快捷键冲突测试。
- 最近历史快捷键触发 slot 1/2 的粘贴目标测试。
- 历史不足时不粘贴测试。
- 托盘菜单构建测试，验证菜单项不包含“打开 ClipVault”。
- 前端设置页测试，验证新增分组、位置文案、录制和启用开关。
- tauriApi 测试，验证新增 commands 调用。

第二阶段自动化测试：

- 更新状态转换测试：idle、checking、available、downloading、installing、error。
- 无更新时不弹窗测试。
- 有更新时用户拒绝不下载测试。
- 用户确认后调用 download/install/relaunch 测试。

人工验证：

- 在 Windows 桌面真实会话中复制 5 条文本，配置位置 1 和位置 2 快捷键，分别确认粘贴最新和次新内容。
- 复制图片和文件路径，确认对应位置能按现有行为写入剪贴板并粘贴/输出路径。
- 托盘左键打开控制面板，右键显示菜单，菜单没有“打开 ClipVault”。
- 自动更新二期需用测试 Release 验证签名、下载、安装和重启。

## 风险与处理

- `Ctrl+A+1` 不是当前全局热键体系的稳定输入形态。第一阶段不支持序列热键，避免误触和底层 hook 复杂度。
- 最近历史位置必须忽略置顶，否则“位置 1=最新复制内容”的心智会被破坏。
- 快捷键数量增加后冲突概率变高，因此默认不绑定。
- 自动更新会改变发布流程，必须单独验证。签名私钥丢失后会影响已安装客户端继续更新，因此私钥管理要先确定。
