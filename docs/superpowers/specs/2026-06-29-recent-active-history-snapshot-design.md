# 最近活跃历史排序与快捷键快照设计

## 背景

ClipVault 当前在用户复制、粘贴或特殊粘贴历史项时，会更新该历史项的 `last_used_at` 和 `use_count`，但主历史列表仍按 `created_at` 排序。因此，旧历史项即使刚被使用，也不会回到列表顶部。

之前曾尝试让使用过的历史项回到顶部，但它会干扰快捷复制上一条/下一条：快捷导航复用了主列表实时排序，导致某一项被使用后立即改变排序，连续按键时目标项会跳动。这个问题的根因不是“用过回顶部”本身不合理，而是主列表排序和快捷键导航序列共用了一套会实时变化的顺序。

本设计把两类语义拆开：

- 主历史列表展示“最近活跃”的内容。
- 快捷键连续操作使用一个短生命周期快照，避免正在操作的槽位或上一条/下一条目标在按键过程中改变。

## 目标

- 主历史列表中，刚复制、刚粘贴或刚被特殊粘贴的历史项应回到顶部。
- 固定项仍然排在普通项前面。
- 最近历史槽位快捷键表示最近活跃的历史项，而不是只表示最近捕获的新复制项。
- 快捷复制上一条/下一条在连续操作期间保持稳定，不被“使用后置顶”打乱。
- 最近历史槽位快捷键在连续操作期间保持稳定，避免同一个按键短时间内指向不同历史项。

## 非目标

- 不改变数据库字段含义：`created_at` 仍表示历史项首次捕获或创建时间，`last_used_at` 表示从 ClipVault 使用该项的时间。
- 不新增云同步或跨设备状态。
- 不把固定项强行塞入最近历史槽位的最前面。
- 不做“几分钟后再置顶”的延迟置顶。列表置顶应立即发生，稳定性由快捷键快照解决。

## 排序语义

### 主历史列表

主历史列表按以下顺序排序：

1. `is_pinned DESC`
2. `COALESCE(last_used_at, created_at) DESC`
3. `created_at DESC`
4. `id DESC`

效果：

- 固定项始终在普通项前。
- 同一固定分组内，最近活跃项靠前。
- 从未被使用过的项使用 `created_at` 作为活跃时间。
- `created_at` 和 `id` 作为稳定的兜底排序，避免时间相同导致列表抖动。

前端本地 store 的排序规则必须与后端历史查询一致，避免接口返回后又被前端排回旧顺序。

### 最近历史槽位

最近历史槽位快捷键按最近活跃时间取前 N 项，但忽略固定优先级：

1. `COALESCE(last_used_at, created_at) DESC`
2. `created_at DESC`
3. `id DESC`

效果：

- `Ctrl+Alt+1` 表示最近活跃的第 1 项。
- `Ctrl+Alt+2` 表示最近活跃的第 2 项。
- 固定项不会仅因为被固定就抢占槽位。
- 刚复制进来的新内容会进入槽位顶部。
- 刚从历史里粘贴过的旧内容也会进入槽位顶部。

### 捕获时间的保留用途

`created_at` 不再作为主历史列表和最近槽位的唯一“最近”标准，但仍保留两个用途：

- 表示真实捕获/创建顺序。
- 在 `last_used_at` 为空或活跃时间相同的情况下提供稳定排序。

## 快捷键快照

### 需要快照的操作

以下快捷操作需要使用短生命周期快照：

- 快捷复制上一条。
- 快捷复制下一条。
- 最近历史槽位快捷键。

快照用于稳定快捷键操作期间的目标序列，不影响主历史列表立即按最近活跃排序刷新。

### 快照创建

当用户首次触发上述快捷键，且当前没有可复用快照时，后端创建一个快照：

- 快照内容是当时按“最近活跃、忽略固定优先级”排序得到的历史项 ID 列表。
- 快照只保存 ID，不复制完整内容。
- 执行具体粘贴前，再根据 ID 获取最新历史项内容，避免内容编辑后使用旧数据。

### 快照复用

连续快捷操作会复用同一个快照。复用条件：

- 快照没有超过空闲超时时间。
- 快照所属的排序策略一致。
- 快捷操作仍属于同一类历史导航/最近槽位操作。

建议空闲超时时间为 1200ms 到 2000ms。具体实现可以先取 1500ms，足够覆盖连续按键，又不会让槽位长时间停留在旧顺序。

### 快照失效

以下情况应让快照失效：

- 超过空闲超时时间。
- 历史被清空。
- 历史项被删除。
- 应用重启。
- 快照中的目标 ID 已不存在时，重建快照；如果重建后仍没有对应目标，则显示 HUD 提示并停止本次粘贴。

### 行为示例

假设最近活跃顺序为 `A, B, C`。

用户按 `Ctrl+Alt+2`：

- 快照创建为 `A, B, C`。
- 槽位 2 粘贴 `B`。
- `B.last_used_at` 立即更新。
- 主历史列表立即变为 `B, A, C`。
- 但快照仍为 `A, B, C`。

如果用户在快照超时前再次按 `Ctrl+Alt+2`：

- 仍然粘贴 `B`。

如果用户停顿超过超时时间后再按 `Ctrl+Alt+2`：

- 新快照按当前最近活跃顺序创建为 `B, A, C`。
- 槽位 2 粘贴 `A`。

快捷复制上一条/下一条同理：一轮连续操作中的导航顺序稳定，下一轮再按最新排序重新开始。

上一条/下一条需要在快照内维护当前游标。游标只记录当前快照中的位置，不直接依赖主列表的实时索引。快照超时或失效后，下一次快捷操作重新创建快照并按最新最近活跃顺序确定起点。

## 数据流

### 使用历史项

1. 用户通过列表、详情、最近槽位、上一条/下一条或特殊粘贴使用历史项。
2. 后端写入剪贴板或模拟粘贴。
3. 后端调用 `increment_use_stats` 更新 `use_count` 和 `last_used_at`。
4. 前端收到更新后的 item。
5. 前端按最近活跃排序更新主列表。

### 加载历史列表

1. 前端请求历史列表。
2. 后端按主历史列表排序返回数据。
3. 前端 store 使用同一排序规则兜底排序。

### 触发快捷键

1. 后端判断是否存在可复用快照。
2. 如果没有，按最近活跃排序创建 ID 快照。
3. 根据快捷操作解析目标 ID。
4. 读取目标项最新内容并执行粘贴。
5. 更新目标项 `last_used_at`。
6. 保留快照直到超时或被失效条件清理。

## 边界情况

- 固定项：主列表固定项优先，但最近槽位和快捷导航不因固定状态改变顺序。
- 收藏项：收藏只影响清理保护，不影响最近活跃排序。
- 删除项：如果快照目标已删除，应跳过或重建快照，不应粘贴失败后静默误用其他内容。
- 空历史：快捷键显示 HUD 提示，不执行粘贴。
- 相同时间戳：使用 `created_at DESC, id DESC` 保持顺序稳定。
- 图片和文件路径项：排序和快照只依赖元数据 ID，不改变原有内容处理规则。

## 测试要求

后端测试：

- `increment_use_stats` 后，普通历史查询把旧项排到同一固定分组顶部。
- 固定项仍排在普通项前。
- 最近历史槽位查询按 `COALESCE(last_used_at, created_at)` 排序，且忽略固定优先级。
- 快照创建后，即使目标项被使用并更新 `last_used_at`，同一槽位在超时前仍解析到同一 ID。
- 快照超时后重新按最新最近活跃顺序解析。
- 快照目标被删除时会重建快照；仍无法解析目标时显示提示，不会粘贴错误项。
- 上一条/下一条在同一快照内按游标移动，不受主列表实时重排影响。

前端测试：

- store 的 `setItems` 和 `upsertItem` 按最近活跃排序。
- 列表复制旧项后，该项回到当前分组顶部。
- 固定项和普通项排序边界正确。

人工验证：

- 在主界面复制一个旧历史项，它立即回到顶部。
- 连续使用快捷复制上一条/下一条，不会因为刚用过的项置顶而跳乱。
- 连续按同一个最近历史槽位快捷键，短时间内目标稳定。
- 停顿超过快照超时后，槽位按最新最近活跃顺序刷新。

## 发布说明

该改动属于历史排序和快捷键行为修复。发布说明应明确：

- 历史列表现在会把刚使用过的内容提升到顶部。
- 快捷复制上一条/下一条使用短时快照，连续操作更稳定。
- 最近历史槽位表示最近活跃内容，并在连续操作时保持短时稳定。
